| App.tsx layout coexistence (canvas + bottom panel + session area) | ✅ Flex regions independent |

### Findings

**Finding 1 — No reconnect timeout (PTY-04, LOW)**

No code path transitions the 'reconnecting' chip to 'error' if the server is unavailable. The amber pulsing chip will persist indefinitely if the server goes down mid-reconnect. Not a functional break in normal operation — `session-ready` arrives and clears the status correctly. This is a UX gap for offline/server-down scenarios.

**Finding 2 — Auto-seed stale closure (RAWTERM-01, LOW in current implementation)**

`App.tsx` line 289 auto-seed effect reads `rawSessions.length` without `rawSessions` in the dependency array. Currently a non-issue because raw sessions have no localStorage persistence (ephemeral by design, per `[Phase 13-01]` decision). Becomes a latent bug if persistence is added later. Logged as tech debt.

**Finding 3 — Dual `SessionStatus` declaration (informational)**

`SessionStatus` type is defined independently in both `usePty.ts:6` and `useSessionManager.ts:3`. Currently identical. Should be consolidated to one source-of-truth to prevent silent divergence when new statuses are added.

**Finding 4 — cwd staleness in raw sessions on project switch (RAWTERM-05/06, LOW)**

`useRawSessionManager` stores `cwd` at session creation time. If the user disconnects from one project and connects to another in the same session, any restored raw sessions would attempt to spawn PTYs in the old project cwd. Not a failure for typical single-project usage.
