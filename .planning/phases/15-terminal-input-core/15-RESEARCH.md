## Summary

The existing codebase already has all the primitives needed for Phase 15. `Terminal.tsx` creates an xterm.js instance, wires `FitAddon`, and exposes it via `onReady`. `RawTerminalPane.tsx` demonstrates exactly the pattern needed: it calls `terminal.onData(sendInput)` to pipe every raw keystroke directly to the PTY. The `sendInput` function (from `usePty`) sends a `{ type: 'input', data }` WebSocket message that the server feeds into node-pty.

The existing PTY protocol already expects `\r` (CR) for Enter ‚Äî that is what the old Composer appended manually (`value + '\r'`). With an xterm.js input terminal, the user pressing Enter naturally generates `\r`, so no manual appending is needed. Arrow keys generate `\x1b[A`/`\x1b[B`, Ctrl+C generates `\x03`, Ctrl+D generates `\x04`, and Tab generates `\x09` ‚Äî all are produced by xterm.js automatically and forwarded unchanged to the PTY.

The only meaningful work is: (1) create a new `TerminalInput` component that wraps an xterm.js instance in a fixed-height container configured as a compact input strip, (2) wire `terminal.onData(sendInput)` instead of a React `onSend` prop, and (3) replace the drag-resize Composer block in `SessionPane.tsx` with the new fixed-height strip. The FitAddon must be used with care: the input strip has a fixed pixel height, so the terminal must be allowed to compute its column width freely but must not auto-resize its row count (or the row count must be locked to 3‚Äì4).

**Primary recommendation:** Model `TerminalInput` directly on `RawTerminalPane` + `Terminal` but strip out all the PTY connection logic ‚Äî the component only wraps an xterm.js display and wires `terminal.onData(sendInput)`. Everything else (PTY, WebSocket, session management) is unchanged.

### Pattern 1: xterm.js Input Strip (the RawTerminalPane pattern)

**What:** Mount an xterm.js instance in a fixed-height div, call `terminal.onData(sendInput)` to forward all keystrokes to the PTY.

**When to use:** Any time raw terminal input fidelity is needed (arrow keys, control sequences, completion).

**Example (adapted from RawTerminalPane.tsx):**

```typescript
// Source: client/components/RawTerminalPane.tsx (existing)
useEffect(() => {
  if (!terminal) return;
  const disposable = terminal.onData(sendInput);
  return () => disposable.dispose();
}, [terminal, sendInput]);
```

The `sendInput` here is the exact same function from `usePty` / `useSession` that the old Composer also called. No new protocol work is needed.

### Pattern 2: Fixed-height xterm.js instance

**What:** Pass a fixed pixel height to the container div. Do NOT call `fitAddon.fit()` on resize of the window ‚Äî only call it once on mount so it computes column count from the strip width, and lock rows.

**When to use:** Input strip that must not grow (TINPUT-04).

**Example:**

```typescript
// TerminalInput container ‚Äî fixed height, full width
<div
  ref={containerRef}
  style={{ width: '100%', height: INPUT_STRIP_HEIGHT_PX, background: 'var(--bg)' }}
/>
```

After `terminal.open(container)`, call `fitAddon.fit()` once. The terminal will compute cols from the available width and set rows from the fixed pixel height. A 3‚Äì4 line strip at 20px line height ‚âà 60‚Äì80px. Suppress subsequent resize events from propagating to the PTY (the input strip does not own the display terminal's dimensions).

**Exact row count:** At 20px cell height, an 80px container gives 4 rows; a 60px container gives 3 rows. Use 80px for comfort.

### Pattern 3: Theme inheritance

**What:** Reuse the exact same theme object from `Terminal.tsx` in `TerminalInput.tsx`.

**When to use:** Always ‚Äî design system requires visual consistency.

**Example:**

```typescript
// Same theme as Terminal.tsx ‚Äî copy verbatim, including accent-derived cursor color
terminal = new XTerm({
  theme: { background: '#0d1117', foreground: '#c9d1d9', cursor: accent, ... },
  rows: 4,  // lock rows so FitAddon only computes cols
  scrollback: 0,  // input strip does not need scrollback
});
```

Setting `rows` in the XTerm constructor options prevents the FitAddon from changing row count on fit. Set `scrollback: 0` so the strip never scrolls ‚Äî it is an input line, not a buffer.

### Pattern 4: SessionPane integration

**What:** Replace the drag-resize Composer block with a fixed-height TerminalInput strip.

**When to use:** This phase.

**Current SessionPane structure (relevant section):**

```tsx
// Current ‚Äî variable height, drag-resizable Composer
const composerPanel = useDragResize(COMPOSER_DEFAULT, COMPOSER_MIN, 'up');
// ...
<div className="resize-handle--h" onMouseDown={composerPanel.onMouseDown} />
<div className="composer-bottom" style={{ height: composerPanel.width }}>
  <AttachBar ... />
  <div className="composer-area">
    <Composer ref={composerRef} onSend={handleSendInput} ... />
  </div>
</div>
```

**New structure:**

```tsx
// New ‚Äî fixed height strip, no drag-resize
<div className="terminal-input-strip">
  <TerminalInput sendInput={handleSendInput} connected={session.connected} accentHex={accentHex} />
</div>
```

The `composerRef` prop on `SessionPane` exists solely for the old Composer textarea focus trick (used by voice input to auto-focus the textarea after transcription). This wiring must be updated to focus the xterm.js instance instead.

### Anti-Patterns to Avoid

- **Calling fitAddon.fit() in a ResizeObserver loop on the input strip:** This causes row count to change as the panel resizes. Set `rows` in the XTerm constructor to lock it.
- **Forwarding PTY resize events from the input terminal to the server:** Only the display terminal (Terminal.tsx) should send resize messages. The input strip terminal must not call `sendResize`.
- **Two PTY connections:** TerminalInput must NOT create its own `usePty` / WebSocket. It receives `sendInput` as a prop (already connected to the session's PTY).
- **Disposing the input terminal on session disconnect:** Keep the terminal mounted so the user can still type; `connected` prop is used to disable or grey-out but not unmount.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Control sequence generation (Ctrl+C = `\x03`, arrow up = `\x1b[A`) | Custom keydown handler that maps keys to bytes | `terminal.onData()` | xterm.js handles the full VT100/xterm protocol including bracketed paste, compose keys, and OS-specific bindings |
| Terminal rendering (cursor, blinking, selection) | Custom DOM input | xterm.js | xterm already handles all rendering in this project |
| Enter = `\r` mapping | Manually appending `\r` in React | xterm.js `onData` | When the user presses Enter in an xterm.js instance, it naturally produces `\r` ‚Äî the PTY already expects this (confirmed in STATE.md) |

**Key insight:** The only custom logic needed is `useEffect(() => { const d = terminal.onData(sendInput); return () => d.dispose(); }, [terminal, sendInput])`. Everything else is xterm.js.

## Common Pitfalls

### Pitfall 1: FitAddon overriding the fixed row count

**What goes wrong:** After calling `fitAddon.fit()`, xterm recomputes rows from the container height. If the container is taller than expected, you get 5‚Äì6 rows instead of 3‚Äì4.

**Why it happens:** FitAddon computes `Math.floor(height / cellHeight)` ‚Äî if the container has padding or border, the effective height is larger.

**How to avoid:** Set `rows: 4` in the XTerm constructor. FitAddon will then only recompute `cols` on fit, leaving rows alone. Verified: xterm constructor option `rows` takes priority over FitAddon's row calculation.

**Warning signs:** Input strip visually taller than expected, scrollbar appears inside the strip.

### Pitfall 2: Double input ‚Äî xterm.js echoes typed characters AND the PTY echoes them

**What goes wrong:** Typed characters appear twice in the display terminal.

**Why it happens:** xterm.js by default does local echo (shows typed chars in the xterm.js instance), and the PTY also echoes via the `data` event back to Terminal.tsx.

**How to avoid:** The input strip should have `disableStdin: false` (default) to accept keyboard input, but should NOT receive PTY `data` output ‚Äî it is write-only from user to PTY. The display terminal (Terminal.tsx) receives all PTY output. The input terminal only calls `terminal.onData(sendInput)` and never calls `terminal.write()`.

**Additional:** Set `cursorBlink: true` on the input terminal to show the user where they are typing. Set `scrollback: 0`.

### Pitfall 3: Focus competition between display terminal and input terminal

**What goes wrong:** User clicks the display terminal, focus moves there, keystrokes stop reaching the PTY.

**Why it happens:** Terminal.tsx calls `terminal.focus()` on mount (line 85 of Terminal.tsx) and on click (line 141). The display terminal is a full xterm.js instance that also accepts keyboard input ‚Äî if it gets focus, `onData` fires on the display terminal, not the input terminal.

**How to avoid:** The display terminal must NOT call `terminal.onData(sendInput)` in the Claude session (it does not ‚Äî only `RawTerminalPane` does this, and that is separate from the Claude PTY session). However, the display terminal should have `disableStdin: true` set to prevent it from stealing keystrokes when focused. Currently Terminal.tsx does not set `disableStdin` ‚Äî this needs to be added, or at minimum the display terminal's `onData` must be a no-op.

**Warning signs:** Keystrokes appear in the display terminal but are not forwarded to the PTY.

### Pitfall 4: composerRef becoming stale after Composer removal

**What goes wrong:** VoiceBar or other components that hold a ref to the Composer textarea break after the textarea is removed.

**Why it happens:** `SessionPane` accepts a `composerRef` prop of type `React.RefObject<HTMLTextAreaElement | null>`. VoiceBar uses this ref to focus the input after transcription.

**How to avoid:** In Phase 15, keep the `Composer` component present (it is removed in Phase 16). The input strip coexists with the Composer temporarily, or the `composerRef` wiring is updated to call `terminal.focus()` instead. The cleanest Phase 15 approach is to keep the Composer mounted (not visible or shadowed) and add the new input strip ‚Äî Phase 16 removes Composer entirely.

**Warning signs:** Voice transcription completes but focus does not land on the input.

### Pitfall 5: Resize events sent to PTY from input terminal

**What goes wrong:** When the input terminal is fitted, it calls `sendResize` with 4 rows, overriding the display terminal's row count and corrupting Claude's layout.

**Why it happens:** Terminal.tsx's `sendResize` callback is wired from `SessionPane` through `handleSendResize`. If TerminalInput uses the same callback, it will send a resize with `rows: 4`.

**How to avoid:** TerminalInput must NOT receive or call `sendResize`. Its `onReady` callback only stores the terminal reference for `onData` wiring. Do not pass `sendResize` to the input strip component.

## Open Questions

1. **Should `disableStdin: true` be added to the display terminal (Terminal.tsx) globally?**
   - What we know: The display terminal currently accepts keyboard input (no `disableStdin`). RawTerminalPane wires `onData` on the display terminal intentionally (raw shell sessions need it). Claude session panes do NOT wire `onData` on the display terminal ‚Äî that is only done in RawTerminalPane.
   - What's unclear: Whether removing keyboard capture from the display terminal in Claude sessions will cause any regression (e.g., copy/paste shortcuts, scroll shortcuts).
   - Recommendation: Add `disableStdin` as a prop on `Terminal.tsx` (default `false`) and pass `disableStdin={true}` from `SessionPane` for Claude sessions. RawTerminalPane continues to pass `disableStdin={false}` (or omit it).

2. **How does Phase 16's slash command overlay wire into the xterm.js input?**
   - What we know: The old SlashMenu detects `/` by watching the textarea's `onChange` event. xterm.js has no `onChange` ‚Äî it produces raw characters via `onData`.
   - What's unclear: Whether Phase 15 should add any infrastructure for Phase 16 slash detection (e.g., a local buffer of current line contents), or leave that fully to Phase 16.
   - Recommendation: Phase 15 is purely the input strip + PTY wiring. Phase 16 will need to intercept `onData` before forwarding to `sendInput` to detect slash commands. This is a Phase 16 concern.

3. **VoiceBar focus after transcription**
   - What we know: `composerRef` is a `RefObject<HTMLTextAreaElement | null>`. Voice input calls `composerRef.current?.focus()` after transcription to return focus.
   - What's unclear: Whether the ref should be changed to `RefObject<{ focus: () => void } | null>` (a generic focusable ref) or whether the xterm instance should be stored separately.
   - Recommendation: In Phase 15, expose a `focus()` method from `TerminalInput` via a forwarded ref (`useImperativeHandle`) so the existing `composerRef`-style wiring can call `focus()` without knowing the underlying type. This avoids touching VoiceBar in Phase 15.
